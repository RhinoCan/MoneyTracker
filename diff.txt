+## Aria testing
+The good news is that aria testing doesn't need to be a separate major effort — it can be layered into what you're already planning:
+For Vitest unit tests, you can use @testing-library/vue which has built-in accessibility queries (getByRole, getByLabelText, etc.) that encourage aria-correct markup by design. Just writing your tests using those queries rather than class selectors or IDs naturally enforces good aria hygiene.
+For Playwright e2e tests, there's @axe-core/playwright (or just axe-core directly) which you can run against each page/view to get a full accessibility audit including aria issues, contrast ratios, missing labels, and so on. It's just a few extra lines per test.
+So the plan would be:
+
+Use @testing-library/vue queries in Vitest to enforce aria-correct component markup
+Add axe-core scans in Playwright to catch page-level issues
+
+Neither requires a separate test suite — they fold into the existing testing work. The main thing to think about now while eyeballing is making sure your Vuetify components have proper labels, that interactive elements have meaningful roles, and that dynamic content changes (like toast notifications) are announced via aria-live regions. Vuetify handles a lot of this automatically but it's worth verifying.
